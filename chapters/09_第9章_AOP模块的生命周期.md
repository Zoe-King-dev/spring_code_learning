# 第9章 AOP 模块的生命周期

本章从 `@EnableAspectJAutoProxy` 注解出发，详细探讨注解驱动 AOP 底层的支撑组件，并针对核心组件 `AnnotationAwareAspectJAutoProxyCreator` 的初始化时机、作用机制进行深入剖析。AOP 代理对象的工作离不开增强器、代理对象执行链等核心组件的支撑，掌握核心组件的工作时机和作用机制可以更容易地理解 AOP 的工作原理。

## 9.1 @EnableAspectJAutoProxy 注解

在 Spring Boot 项目中使用 AOP 功能时，首先需要在配置类上标注 `@EnableAspectJAutoProxy` 注解来开启 AOP 功能。该注解的核心作用是向容器中注册一个 `AnnotationAwareAspectJAutoProxyCreator` Bean 后置处理器，用于处理切面类和生成代理对象。

## 9.2 AnnotationAwareAspectJAutoProxyCreator

### 9.2.1 类继承结构

`AnnotationAwareAspectJAutoProxyCreator` 继承自 `AspectJAutoProxyCreator`，最终实现了 `BeanPostProcessor` 接口。它的类继承层次如下：

```
AnnotationAwareAspectJAutoProxyCreator
    └── AspectJAutoProxyCreator
        └── AbstractAutoProxyCreator
            └── ProxyConfig
            └── BeanPostProcessor
```

### 9.2.2 初始化时机

作为 `BeanPostProcessor` 的实现类，`AnnotationAwareAspectJAutoProxyCreator` 在容器刷新阶段就会注册到容器中，并在每个 Bean 初始化完成后进行处理。

### 9.2.3 作用时机

`AnnotationAwareAspectJAutoProxyCreator` 实现了 `postProcessAfterInitialization` 方法，在 Bean 对象初始化完成后会尝试创建代理对象。其核心调用链如下：

1. `postProcessAfterInitialization` 被容器回调
2. 调用 `wrapIfNecessary` 方法判断是否需要生成代理
3. 如需代理，则调用 `createProxy` 方法创建代理对象

## 9.3 Advisor 和 Aspect 的收集

### 9.3.1 收集增强器的逻辑

在 `wrapIfNecessary` 方法被调用后，会通过 `getAdvicesAndAdvisorsForBean` 方法获取适用于当前 Bean 的增强器。该方法内部调用 `findEligibleAdvisors`，其过程分为三个步骤：

1. **获取所有候选增强器**：`findCandidateAdvisors`
2. **筛选适用于当前 Bean 的增强器**：`findAdvisorsThatCanApply`
3. **添加额外增强器**：`extendAdvisors`

### 9.3.2 收集 Spring Framework 原生 Advisors

`findCandidateAdvisors` 方法会扫描容器中所有实现了 `Advisor` 接口的 Bean，将它们作为候选增强器。

### 9.3.3 解析 Aspect 切面类

对于 `@Aspect` 注解标注的切面类，`AnnotationAwareAspectJAutoProxyCreator` 会通过 `BeanFactoryAspectJAdvisorsBuilder` 进行解析，将通知方法封装为 `Advisor` 对象。

## 9.4 TargetSource 的设计

### 9.4.1 TargetSource 接口设计

`TargetSource` 是 Spring AOP 框架中用于获取目标对象的接口，其核心方法如下：

```java
public interface TargetSource extends TargetClassAware {

    // 获取目标对象
    TargetSource getTarget() throws Exception;

    // 释放目标对象
    void releaseTarget(Object target) throws Exception;

    // 判断目标对象是否为静态
    boolean isStatic();

    // 获取目标对象的类型
    @Override
    Class<?> getTargetClass();
}
```

### 9.4.2 为什么需要 TargetSource

在 AOP 代理对象执行方法时，需要从 `TargetSource` 中获取目标对象。使用 `TargetSource` 的好处是：

1. **延迟加载**：目标对象可以在需要时才创建
2. **对象池化**：可以复用目标对象，减少创建开销
3. **热替换**：可以在运行时动态替换目标对象

### 9.4.3 静态与动态 TargetSource

对于 Spring Framework 而言：

- **单实例 Bean 对象**：在 ApplicationContext 中只有一个实例，属于"静态 Bean 对象"
- **原型 Bean 对象**：每次获取都会得到一个全新的实例，属于"非静态 Bean 对象"

`TargetSource` 接口中的 `isStatic()` 方法用于区分这两种情况。

### 9.4.4 Spring Framework 中提供的 TargetSource

Spring Framework 针对不同场景预设了几个 `TargetSource` 实现：

| 实现类 | 说明 |
|--------|------|
| `SingletonTargetSource` | 每次调用 `getTarget` 都返回同一个目标 Bean 对象 |
| `PrototypeTargetSource` | 每次调用 `getTarget` 都从 BeanFactory 中创建一个全新的 Bean 对象（被包装的 Bean 必须是原型 Bean） |
| `CommonsPool2TargetSource` | 内部维护一个对象池，使用 Apache 的 `ObjectPool` |
| `ThreadLocalTargetSource` | 每次调用 `getTarget` 都从当前线程中提取目标对象（被包装的 Bean 必须是原型 Bean） |
| `HotSwappableTargetSource` | 内部维护一个可以热替换的目标对象引用，提供了线程安全的 `swap` 方法 |

## 9.5 代理对象生成的核心：wrapIfNecessary

在 9.2.3 节中已经了解到，最终 Bean 对象的初始化会被 `BeanPostProcessor` 的 `postProcessAfterInitialization` 方法处理，进入 `AnnotationAwareAspectJAutoProxyCreator` 的 `postProcessAfterInitialization` 方法中，它又会调用 `wrapIfNecessary` 方法来尝试创建代理。

### 9.5.1 wrapIfNecessary 方法逻辑

`wrapIfNecessary` 方法的完整逻辑如下：

```java
protected Object wrapIfNecessary(Object bean, String beanName, Object cacheKey) {
    // 判断是否是不需要增强的 Bean 对象
    if (StringUtils.hasLength(beanName) && this.targetSourcedBeans.contains(beanName)) {
        return bean;
    }

    // 已经判断过不需要代理，直接返回
    if (Boolean.FALSE.equals(this.advisedBeans.get(cacheKey))) {
        return bean;
    }

    // 判断是否是基础设施类或应该跳过增强的类
    if (isInfrastructureClass(bean.getClass()) || shouldSkip(bean.getClass(), beanName)) {
        this.advisedBeans.put(cacheKey, Boolean.FALSE);
        return bean;
    }

    // 如果上面的判断条件都不成立，则决定是否需要创建代理对象
    Object[] specificInterceptors = getAdvicesAndAdvisorsForBean(bean.getClass(), beanName, null);

    if (specificInterceptors != DO_NOT_PROXY) {
        this.advisedBeans.put(cacheKey, Boolean.TRUE);
        // 创建代理对象
        Object proxy = createProxy(bean.getClass(), beanName, specificInterceptors,
                new SingletonTargetSource(bean));
        this.proxyTypes.put(cacheKey, proxy.getClass());
        return proxy;
    }

    this.advisedBeans.put(cacheKey, Boolean.FALSE);
    return bean;
}
```

### 9.5.2 getAdvicesAndAdvisorsForBean 方法

该方法根据当前正在初始化的 Bean 对象，匹配可供织入通知的增强器：

```java
protected Object[] getAdvicesAndAdvisorsForBean(
        Class<?> beanClass, String beanName, @Nullable TargetSource targetSource) {
    List<Advisor> advisors = findEligibleAdvisors(beanClass, beanName);
    if (advisors.isEmpty()) {
        return DO_NOT_PROXY;
    }
    return advisors.toArray();
}
```

`findEligibleAdvisors` 方法分为三个步骤：

```java
protected List<Advisor> findEligibleAdvisors(Class<?> beanClass, String beanName) {
    // 1. 获取所有增强器
    List<Advisor> candidateAdvisors = findCandidateAdvisors();

    // 2. 筛选出可以切入当前 Bean 对象的增强器
    List<Advisor> eligibleAdvisors = findAdvisorsThatCanApply(candidateAdvisors, beanClass, beanName);

    // 3. 添加额外的增强器
    extendAdvisors(eligibleAdvisors);

    if (!eligibleAdvisors.isEmpty()) {
        // 增强器排序
        eligibleAdvisors = sortAdvisors(eligibleAdvisors);
    }
    return eligibleAdvisors;
}
```

### 9.5.3 findAdvisorsThatCanApply 方法

该方法用于匹配可以切入当前 Bean 对象的增强器：

```java
protected List<Advisor> findAdvisorsThatCanApply(
        List<Advisor> candidateAdvisors, Class<?> beanClass, String beanName) {
    ProxyCreationContext.setCurrentProxiedBeanName(beanName);
    try {
        return AopUtils.findAdvisorsThatCanApply(candidateAdvisors, beanClass);
    } finally {
        ProxyCreationContext.setCurrentProxiedBeanName(null);
    }
}
```

在 `AopUtils.findAdvisorsThatCanApply` 中，核心逻辑分为两部分：

1. **引介增强器匹配**：使用类级过滤器匹配
2. **普通方法增强器匹配**：使用 `PointcutAdvisor` 进行切点表达式匹配

### 9.5.4 extendAdvisors 添加额外增强器

`extendAdvisors` 方法会检查当前增强器列表中是否存在 AspectJ 类型的增强器，如果存在，则会在列表最前端添加一个 `ExposeInvocationInterceptor.ADVISOR`：

```java
protected void extendAdvisors(List<Advisor> candidateAdvisors) {
    AspectJProxyUtils.makeAdvisorChainAspectJCapableIfNecessary(candidateAdvisors);
}

// AspectJProxyUtils
public static boolean makeAdvisorChainAspectJCapableIfNecessary(List<Advisor> advisors) {
    if (!advisors.isEmpty()) {
        boolean foundAspectJAdvice = false;
        for (Advisor advisor : advisors) {
            if (isAspectJAdvice(advisor)) {
                foundAspectJAdvice = true;
                break;
            }
        }
        // 如果发现有 AspectJ 封装的增强器，则添加一个 ExposeInvocationInterceptor
        if (foundAspectJAdvice && !advisors.contains(ExposeInvocationInterceptor.ADVISOR)) {
            advisors.add(0, ExposeInvocationInterceptor.ADVISOR);
            return true;
        }
    }
    return false;
}
```

### 9.5.5 createProxy 方法

创建代理对象的逻辑如下：

```java
protected Object createProxy(Class<?> beanClass, @Nullable String beanName,
        @Nullable Object[] specificInterceptors, TargetSource targetSource) {

    // 代理工厂的初始化
    ProxyFactory proxyFactory = new ProxyFactory();
    proxyFactory.copyFrom(this);

    // 根据 AOP 的设计，决定是否强制使用 Cglib
    if (!proxyFactory.isProxyTargetClass()) {
        if (shouldProxyTargetClass(beanClass, beanName)) {
            proxyFactory.setProxyTargetClass(true);
        } else {
            // 解析被代理 Bean 对象所属类的所有实现的接口
            evaluateProxyInterfaces(beanClass, proxyFactory);
        }
    }

    // 构造整合所有增强器
    Advisor[] advisors = buildAdvisors(beanName, specificInterceptors);
    proxyFactory.addAdvisors(advisors);
    proxyFactory.setTargetSource(targetSource);
    customizeProxyFactory(proxyFactory);

    // 创建代理对象
    return proxyFactory.getProxy(getProxyClassLoader());
}
```

### 9.5.6 buildAdvisors 方法

`buildAdvisors` 方法整合所有需要织入目标对象的通知增强器：

```java
protected Advisor[] buildAdvisors(@Nullable String beanName,
        @Nullable Object[] specificInterceptors) {
    // 此处是为适配 Spring 原生 AOP 的 MethodInterceptor
    Advisor[] commonInterceptors = resolveInterceptorNames();

    List<Object> allInterceptors = new ArrayList<>();
    if (specificInterceptors != null) {
        allInterceptors.addAll(Arrays.asList(specificInterceptors));
    }

    // 组合原生的方法拦截器，共同作为 AOP 的通知织入
    if (commonInterceptors.length > 0) {
        if (this.applyCommonInterceptorsFirst) {
            allInterceptors.addAll(0, Arrays.asList(commonInterceptors));
        } else {
            allInterceptors.addAll(Arrays.asList(commonInterceptors));
        }
    }

    Advisor[] advisors = new Advisor[allInterceptors.size()];
    for (int i = 0; i < allInterceptors.size(); i++) {
        // 此处有一个原生 AOP 的适配动作
        advisors[i] = this.advisorAdapterRegistry.wrap(allInterceptors.get(i));
    }
    return advisors;
}
```

### 9.5.7 动态代理方式的选择

进入 `DefaultAopProxyFactory` 的 `createAopProxy` 方法，可以看到底层使用 JDK 动态代理还是 Cglib 动态代理的选择逻辑：

```java
public AopProxy createAopProxy(AdvisedSupport config) throws AopConfigException {
    if (config.isOptimize() || config.isProxyTargetClass()
            || hasNoUserSuppliedProxyInterfaces(config)) {
        Class<?> targetClass = config.getTargetClass();
        if (targetClass == null) {
            // 如果要代理的本身就是接口，或者已经是被 JDK 动态代理的代理对象，则使用 JDK 动态代理
            if (targetClass.isInterface() || Proxy.isProxyClass(targetClass)) {
                return new JdkDynamicAopProxy(config);
            }
            // 否则使用 Cglib 动态代理
            return new ObjenesisCglibAopProxy(config);
        } else {
            return new JdkDynamicAopProxy(config);
        }
    } else {
        return new JdkDynamicAopProxy(config);
    }
}
```

## 9.6 代理对象的底层执行逻辑

IOC 容器刷新完毕后，接下来探讨程序运行期间，被 AOP 代理的对象执行被增强的方法时内部的调用机制。

### 9.6.1 CglibAopProxy 的执行流程

当调用代理对象的方法时，会进入 `CglibAopProxy` 的内部类 `DynamicAdvisedInterceptor` 的 `intercept` 方法：

```java
public Object intercept(Object proxy, Method method, Object[] args, MethodProxy methodProxy)
        throws Throwable {

    Object oldProxy = null;
    boolean setProxyContext = false;
    Object target = null;
    TargetSource targetSource = this.advised.getTargetSource();
    try {
        // 如果在 @EnableAspectJAutoProxy 注解上配置了 exposeProxy 属性为 true，
        // 则会把当前代理对象放入 AOP 上下文中
        if (this.advised.exposeProxy) {
            oldProxy = AopContext.setCurrentProxy(proxy);
            setProxyContext = true;
        }

        // 从 TargetSource 中提取出目标对象
        target = targetSource.getTarget();
        Class<?> targetClass = (target != null ? target.getClass() : null);

        // 根据当前执行的方法，获取要执行的增强器，并以列表返回（链的思想）
        List<Object> chain = this.advised.getInterceptorsAndDynamicInterceptionAdvice(
                method, targetClass);

        Object retVal;

        // 如果没有要执行的增强器，则直接执行目标方法
        if (chain.isEmpty() && Modifier.isPublic(method.getModifiers())) {
            Object[] argsToUse = AopProxyUtils.adaptArgumentsIfNecessary(method, args);
            retVal = methodProxy.invoke(target, argsToUse);
        } else {
            // 否则，构造增强器链，执行增强器的逻辑
            retVal = new CglibMethodInvocation(proxy, target, method, args,
                    targetClass, chain, methodProxy).proceed();
        }

        retVal = processReturnType(proxy, target, method, retVal);
        return retVal;
    } finally {
        if (setProxyContext) {
            AopContext.setCurrentProxy(oldProxy);
        }
    }
}
```

### 9.6.2 获取增强器链

`getInterceptorsAndDynamicInterceptionAdvice` 方法使用了经典的缓存设计，当方法第一次调用完成后，当前方法涉及的增强器链就会被缓存：

```java
public List<Object> getInterceptorsAndDynamicInterceptionAdvice(Method method,
        @Nullable Class<?> targetClass) {
    MethodCacheKey cacheKey = new MethodCacheKey(method);
    List<Object> cached = this.methodCache.get(cacheKey);
    if (cached == null) {
        // 核心逻辑
        cached = this.advisorChainFactory.getInterceptorsAndDynamicInterceptionAdvice(
                this, method, targetClass);
        this.methodCache.put(cacheKey, cached);
    }
    return cached;
}
```

在 `advisorChainFactory.getInterceptorsAndDynamicInterceptionAdvice` 中：

1. **前置准备**：初始化 `AdvisorAdapterRegistry`，用于将 AspectJ 类型的增强器转换为 `MethodInterceptor`

2. **匹配增强器**：使用 `MethodMatcher` 进行切点表达式匹配

3. **动态匹配器处理**：如果 `MethodMatcher` 是动态匹配器，会将参数一并纳入匹配规则

4. **其他增强器处理**：处理引介增强器及其他特殊类型的增强器

### 9.6.3 执行增强器链

`CglibMethodInvocation` 继承自 `ReflectiveMethodInvocation`，最终的 `proceed` 方法实现在 `ReflectiveMethodInvocation` 中：

```java
public Object proceed() throws Throwable {
    // 增强器全部执行完毕后，会执行目标方法
    if (this.currentInterceptorIndex == this.interceptorsAndDynamicMethodMatchers.size() - 1) {
        return invokeJoinpoint();
    }

    // 依次取出增强器封装的拦截器并执行
    Object interceptorOrInterceptionAdvice =
            this.interceptorsAndDynamicMethodMatchers.get(++this.currentInterceptorIndex);

    if (interceptorOrInterceptionAdvice instanceof InterceptorAndDynamicMethodMatcher) {
        // 此处是动态匹配器构造的特殊逻辑
        InterceptorAndDynamicMethodMatcher dm =
                (InterceptorAndDynamicMethodMatcher) interceptorOrInterceptionAdvice;
        Class<?> targetClass = (this.targetClass != null
                ? this.targetClass : this.method.getDeclaringClass());

        // 动态匹配器必须将参数一并纳入匹配规则中
        if (dm.methodMatcher.matches(this.method, targetClass, this.arguments)) {
            return dm.interceptor.invoke(this);
        } else {
            // 动态匹配器在匹配方法调用的参数值列表时发现匹配不上，则这个增强器不会执行
            return proceed();
        }
    } else {
        // 此处会调用增强器的逻辑
        return ((MethodInterceptor) interceptorOrInterceptionAdvice).invoke(this);
    }
}
```

### 9.6.4 JDK 动态代理的执行底层

JDK 动态代理的 `InvocationHandler` 是 `JdkDynamicAopProxy`，其核心 `invoke` 方法如下：

```java
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    Object oldProxy = null;
    boolean setProxyContext = false;
    TargetSource targetSource = this.advised.targetSource;
    Object target = null;

    try {
        // equals 方法不代理
        if (!this.equalsDefined && AopUtils.isEqualsMethod(method)) {
            return equals(args[0]);
        } else if (!this.hashCodeDefined && AopUtils.isHashCodeMethod(method)) {
            // hashcode 方法不代理
            return hashCode();
        } else if (method.getDeclaringClass() == DecoratingProxy.class) {
            // 方法来自 DecoratingProxy 接口的也不代理
            return AopProxyUtils.ultimateTargetClass(this.advised);
        } else if (!this.advised.opaque && method.getDeclaringClass().isInterface()
                && method.getDeclaringClass().isAssignableFrom(Advised.class)) {
            // 目标对象本身实现了 Advised 接口的也不代理
            return AopUtils.invokeJoinpointUsingReflection(this.advised, method, args);
        }

        Object retVal;
        if (this.advised.exposeProxy) {
            oldProxy = AopContext.setCurrentProxy(proxy);
            setProxyContext = true;
        }

        target = targetSource.getTarget();
        Class<?> targetClass = (target != null ? target.getClass() : null);

        // 根据当前执行的方法获取要执行的增强器，并以列表返回（链的思想）
        List<Object> chain = this.advised.getInterceptorsAndDynamicInterceptionAdvice(
                method, targetClass);

        if (chain.isEmpty()) {
            Object[] argsToUse = AopProxyUtils.adaptArgumentsIfNecessary(method, args);
            retVal = AopUtils.invokeJoinpointUsingReflection(target, method, argsToUse);
        } else {
            MethodInvocation invocation = new ReflectiveMethodInvocation(proxy, target,
                    args, targetClass, chain);
            // 构造增强器链，执行增强器的逻辑
            retVal = invocation.proceed();
        }

        // 返回值的处理
        return retVal;
    } finally {
        if (setProxyContext) {
            AopContext.setCurrentProxy(oldProxy);
        }
    }
}
```

## 9.7 AspectJ 中通知的底层实现

本节介绍 AspectJ 中 5 种声明式通知方法的底层执行逻辑。

### 9.7.1 @Before 前置通知

先执行前置通知，再执行目标方法：

```java
public class MethodBeforeAdviceInterceptor implements MethodInterceptor, Serializable {

    private MethodBeforeAdvice advice;

    @Override
    public Object invoke(MethodInvocation mi) throws Throwable {
        this.advice.before(mi.getMethod(), mi.getArguments(), mi.getThis());
        return mi.proceed();
    }
}
```

### 9.7.2 @After 后置通知

执行目标方法后，在 finally 中执行后置方法：

```java
public class AspectJAfterAdvice extends AbstractAspectJAdvice
        implements MethodInterceptor, AfterAdvice, Serializable {

    @Override
    public Object invoke(MethodInvocation mi) throws Throwable {
        try {
            return mi.proceed();
        } finally {
            invokeAdviceMethod(getJoinPointMatch(), null, null);
        }
    }
}
```

### 9.7.3 @AfterReturning 返回通知

返回值后置处理中不设置 try-catch 结构，说明不出现任何异常时才会触发该后置通知：

```java
public class AfterReturningAdviceInterceptor implements MethodInterceptor,
        AfterAdvice, Serializable {

    private final AfterReturningAdvice advice;

    @Override
    public Object invoke(MethodInvocation mi) throws Throwable {
        Object retVal = mi.proceed();
        this.advice.afterReturning(retVal, mi.getMethod(), mi.getArguments(), mi.getThis());
        return retVal;
    }
}
```

### 9.7.4 @AfterThrowing 异常通知

出现异常时进入该后置通知，因为设置了 try-catch 结构，所以在 catch 中根据是否标注了异常通知进行相应的后置处理：

```java
public class AspectJAfterThrowingAdvice extends AbstractAspectJAdvice
        implements MethodInterceptor, AfterAdvice, Serializable {

    @Override
    public Object invoke(MethodInvocation mi) throws Throwable {
        try {
            return mi.proceed();
        } catch (Throwable ex) {
            if (shouldInvokeOnThrowing(ex)) {
                invokeAdviceMethod(getJoinPointMatch(), null, ex);
            }
            throw ex;
        }
    }
}
```

### 9.7.5 @Around 环绕通知

通过构建 `ProceedingJoinPoint` 对象，直接以参数形式传入通知方法的参数中（如果需要的话），反射执行通知方法：

```java
public class AspectJAroundAdvice extends AbstractAspectJAdvice
        implements MethodInterceptor, Serializable {

    @Override
    public Object invoke(MethodInvocation mi) throws Throwable {
        if (!(mi instanceof ProxyMethodInvocation)) {
            throw new IllegalStateException("MethodInvocation is not a Spring ProxyMethodInvocation: " + mi);
        }
        ProxyMethodInvocation pmi = (ProxyMethodInvocation) mi;
        ProceedingJoinPoint pjp = lazyGetProceedingJoinPoint(pmi);
        JoinPointMatch jpm = getJoinPointMatch(pmi);
        return invokeAdviceMethod(pjp, jpm, null, null);
    }
}
```

## 9.8 AOP 通知的执行顺序对比

不同的 Spring Framework 版本中 AOP 通知的执行顺序是有差异的。本节通过一个示例来探究该差异。

### 9.8.1 测试代码编写

为了测试出尽可能多的通知执行顺序差别，在编写 Aspect 切面类时需要覆盖所有类型的通知：

```java
@Component
public class ServiceAspect {

    @Before("execution(public * com.linkedbear.springboot.service.*.*(..))")
    public void beforePrint() {
        System.out.println("Service Aspect before advice run....");
    }

    @After("execution(public * com.linkedbear.springboot.service.*.*(..))")
    public void afterPrint() {
        System.out.println("Service Aspect after advice run ...");
    }

    @AfterReturning("execution(public * com.linkedbear.springboot.service.*.*(..))")
    public void afterReturningPrint() {
        System.out.println("Service Aspect afterReturning advice run .....");
    }

    @AfterThrowing("execution(public * com.linkedbear.springboot.service.*.*(..))")
    public void afterThrowingPrint() {
        System.out.println("Service Aspect afterThrowing advice run .....");
    }

    @Around("execution(public * com.linkedbear.springboot.service.*.*(..))")
    public Object aroundPrint(ProceedingJoinPoint joinPoint) throws Throwable {
        System.out.println("Service Aspect around before run ....");
        Object ret = joinPoint.proceed();
        System.out.println("Service Aspect around after run ......");
        return ret;
    }
}
```

### 9.8.2 Spring Framework 5.x 的顺序

运行应用程序后，控制台输出如下：

```
Service Aspect around before run....
Service Aspect before advice run ..
DemoService save run .
Service Aspect afterReturning advice run ....
Service Aspect after advice run.Service Aspect around after run ...
```

Spring Framework 5.x 中通知的执行顺序：

1. 环绕通知中 `joinPoint.proceed()` 之前的逻辑
2. 前置通知
3. 目标对象的目标方法
4. 返回通知
5. 后置通知
6. 环绕通知中 `joinPoint.proceed()` 之后的逻辑

### 9.8.3 Spring Framework 4.x 的顺序

Spring Framework 4.x 中通知的执行顺序：

```
Service Aspect around before run .....
Service Aspect before advice run .....
DemoService save run .....
Service Aspect around after run .....
Service Aspect after advice run .....
Service Aspect afterReturning advice run .....
```

1. 环绕通知中 `joinPoint.proceed()` 之前的逻辑
2. 前置通知
3. 目标对象的目标方法
4. 环绕通知中 `joinPoint.proceed()` 之后的逻辑
5. 后置通知
6. 返回通知

### 9.8.4 重点总结

Spring Framework 5.x 与 4.x 中 AOP 通知顺序的不同之处在于**环绕通知的范围**：

- Spring Framework 5.x 中环绕通知会包含其他所有通知
- Spring Framework 4.x 中环绕通知只会包含前置通知，后置通知和返回通知会在环绕通知执行完成后执行

## 9.9 小结

本章从 AOP 的开启注解 `@EnableAspectJAutoProxy` 出发，详细探讨了注解驱动 AOP 底层的支撑组件，并针对核心组件 `AnnotationAwareAspectJAutoProxyCreator` 的初始化时机、作用机制进行了深入剖析。

AOP 代理对象的工作离不开增强器、代理对象执行链等核心组件的支撑，掌握核心组件的工作时机和作用机制可以更容易地理解 AOP 的工作原理。
